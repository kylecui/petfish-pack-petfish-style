# Formatting Rules

## 中英混排规则

本skill默认采用紧凑工程写作风格，而不是出版物式中英空格规范。

### 默认去除空格

-中文 + 英文技术词：`使用Git管理代码`
-英文技术词 + 中文名词：`API接口`
-英文技术词 + 中文动词：`Webhook挂载`
-英文缩写 + 中文名词：`XDP程序`、`TLS握手`

### 斜杠分隔词

用`/`分隔的英文词组与中文相邻时，去除`/`两侧及中英之间的空格：

- ✅ `根据API/CLI/SDK/配置文件生成文档`
- ❌ `根据 API / CLI / SDK / 配置文件生成文档`
- ✅ `支持HTTP/HTTPS/WebSocket协议`
- ❌ `支持 HTTP / HTTPS / WebSocket 协议`

### 保留空格的情况

-英文是完整短语或句子：`该模块实现了 access control at network layer 的基本能力。`
-英文被引用作为独立对象：`这里的 "Control Plane" 不是普通管理后台。`
-用户明确要求出版排版规范。

## 标点

-中文句子使用中文标点：，。；：
-英文句子使用英文标点。
-不要用过多括号解释。必要解释优先放入下一句。

## 引号

引号只用于：

-引用原文
-特定术语首次出现
-需要强调术语边界

不要使用引号制造语气。

## 标题格式规范

### 编号与空格

- 节编号后必须跟半角空格：`### 1.1 标题` 而非 `### 1.1标题`
- 多级编号同理：`#### 1.1.1 子节` 而非 `#### 1.1.1子节`
- 中文序号后必须跟半角空格：`## 第一节 概述` 而非 `## 第一节概述`
- 适用的中文序号：第X节/部分/章/篇/模块/单元

### 全角空格

- 标题中禁止使用全角空格（U+3000），统一使用半角空格

### 编号层级

- H2 (`##`) 用于章节/模块级标题
- H3 (`###`) 用于 `X.Y` 格式的子节
- H4 (`####`) 用于 `X.Y.Z` 格式的子子节
- 避免跳级（如 H1 直接接 H3）

### 标点

- 标题末尾不加句号
- 冒号仅在标题本身包含冒号时使用（如 "FAQ:"、"Q&A:"）
- 标题末尾不加顿号或逗号

### 检测工具

- `normalize_text.py --file input.md --output output.md` 自动修复标题空格
- `style_check.py --file input.md` 检测标题格式问题
- `audit_headings.py --files *.md` 列出所有标题格式变体（修复前枚举、修复后验证零残留）

## 边界情况判定流程

遇到不确定是否加空格的情况时，按以下顺序判断：

1. **是否为技术术语？** 如果是产品名、协议名、框架名、工具名 → 紧凑（不加空格）
2. **是否构成紧凑技术表达？** 如果"英文词+中文"构成一个完整的技术概念 → 紧凑
3. **英文是否为完整句子或从句？** 如果是 → 保留空格
4. **英文是否被引号引用为独立对象？** 如果是 → 保留空格
5. **用户是否明确要求出版排版？** 如果是 → 按出版规范处理
6. **以上都不适用？** → 默认紧凑

## normalize_text.py 处理范围

`normalize_text.py` 自动处理以下场景：

- 中英文之间的多余空格（保留必要的例外）
- 斜杠分隔词组周围的空格
- 中文标点前后的多余空格
- 标题编号后的缺失空格
- 全角空格（U+3000）替换为半角空格

不自动处理的场景（需人工判断）：
- 英文完整句子中的空格（无法自动区分技术术语和普通英文）
- 引号内的独立术语引用
- 用户指定的出版排版规范

## 常见误判

以下情况容易被误判为"需要加空格"，实际应该紧凑：

- `grep搜索`（不是 `grep 搜索`）— 工具名+中文动词
- `sed替换`（不是 `sed 替换`）— 工具名+中文动词
- `awk处理`（不是 `awk 处理`）— 工具名+中文动词
- `curl请求`（不是 `curl 请求`）— 工具名+中文名词
- `wget下载`（不是 `wget 下载`）— 工具名+中文动词
- `pip安装`（不是 `pip 安装`）— 包管理器+中文动词
- `npm打包`（不是 `npm 打包`）— 包管理器+中文动词
- `brew安装`（不是 `brew 安装`）— 包管理器+中文动词
- `apt升级`（不是 `apt 升级`）— 包管理器+中文动词
- `yum安装`（不是 `yum 安装`）— 包管理器+中文动词

## 数字与单位

- 中文 + 阿拉伯数字 + 量词：紧凑写。如 `延迟降低30%`，而非 `延迟降低 30%`
- 中文 + 纯英文单位：紧凑写。如 `带宽为10Gbps`，而非 `带宽为 10 Gbps`
- 中文 + 度量衡缩写：紧凑写。如 `文件大小不超过5MB`，而非 `文件大小不超过 5 MB`
- 大数使用千分位：`1,024` 而非 `1024`（当数字超过4位时）
- 百分比与中文之间不空格：`准确率达到95.3%`

### 日期与时间

- ISO 8601格式优先：`2026-06-17T14:30:00+08:00`
- 简写日期：`2026-06-17`，而非 `2026 年 6 月 17 日`（技术文档中）
- 时间段用连字符：`14:30-15:00`
- 毫秒标注紧凑：`响应时间为120ms`

## 文件名与路径

- 文件扩展名与中文紧凑：`配置文件为config.yaml`，而非 `配置文件为 config.yaml`
- 路径中的斜杠不加空格：`路径为/etc/nginx/conf.d/default.conf`
- 代码标识符用反引号包裹后与中文紧凑：`函数返回值通过`resolve_endpoint()`处理`

## 代码块与内联代码

### 内联代码规则

- 内联代码与相邻中文之间不空格：`调用`fetch_data()`函数`
- 内联代码与相邻英文之间空一格：`Call the `fetch_data()` function`
- 内联代码出现在句末时，句号紧跟：`返回值为`None`。`

### 代码块规则

- 代码块前后各空一行
- 代码块标注语言类型（如python、bash、yaml），不要使用无标签的代码块
- 代码块内的缩进保持原样，不额外加空格
- 代码块注释用对应语言的注释语法，不混用中文标点

## 常见错误模式速查表

| 错误写法 | 正确写法 | 原因 |
|---|---|---|
| `使用 Docker 部署` | `使用Docker部署` | 中文名词+英文技术词紧凑 |
| `配置 ENV 变量` | `配置ENV变量` | 环境变量缩写紧凑 |
| `端口为 8080` | `端口为8080` | 数字与中文紧凑 |
| `大小约 16 GB` | `大小约16GB` | 数字+单位与中文紧凑 |
| `版本 >= 1.2` | `版本≥1.2` | 比较运算符紧凑（技术表达） |
| `通过 HTTPS / TLS 加密` | `通过HTTPS/TLS加密` | 斜杠分隔紧凑 |
| `修改 .env 文件` | `修改.env文件` | 文件名紧凑 |
| `运行 npm install 后` | `运行npm install后` | 命令紧凑（但命令内部空格保留） |

## 排版验证清单

在提交或发布前，逐项检查：

1. 所有中英文技术词交界处是否已去除多余空格
2. 斜杠分隔词组两侧是否已去除空格
3. 数字、单位、百分比与中文是否紧凑
4. 文件名、路径、扩展名是否紧凑
5. 内联代码与中文是否紧凑
6. 标题编号后是否有半角空格
7. 标题中是否残留全角空格
8. 标题末尾是否多余标点
9. 标点是否符合中英文混合规则
10. 引号是否仅用于引用和术语边界

运行 `uv run scripts/normalize_text.py --file draft.md --output draft.fixed.md` 可自动修复1-3类问题。运行 `uv run scripts/style_check.py --file draft.fixed.md` 可检测残留问题。
